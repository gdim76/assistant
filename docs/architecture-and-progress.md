# Архитектура решения, техническое задание и прогресс исполнения

## 1. Архитектура решения

### Основные компоненты
- `MainActivity` — стартовая точка приложения, инициализирует репозитории и задает Compose UI.
- `MainScreen` — основной экран пользователя с генерацией урока, статусом, голосовой практикой и переходом к настройкам.
- `VoicePracticeScreen` — экран распознавания речи и голосовой практики.
- `SettingsScreen` — экран настроек LLM провайдера и API ключа.

### Слой данных
- `ProgressDatabase` / `ProgressDao` — локальная база Room для сохранения истории уроков.
- `StudentProfile` / `StudentProfileDao` — локальная память об ученике: базовые данные, цель обучения, стартовый уровень и результат первичной диагностики.
- `SettingsRepository` — хранение настроек приложения через `DataStore Preferences`.
- `LlmRepository` — бизнес-логика генерации урока и сохранения прогресса.

### LLM-слой
- `LlmService` — интерфейс для вызова LLM.
- `MockLlmService` — заглушка для разработки без ключа.
- `OpenAiLlmService` — реализация обращения к OpenAI API.
- `GeminiLlmService` — реализация обращения к Gemini API.
- `LlmServiceFactory` — фабрика выбора провайдера на основе сохраненных настроек.
- `LlmProvider` — перечисление поддерживаемых провайдеров (`GEMINI`, `OPENAI`).

### UI и навигация
- Навигация реализована внутри `MainScreen` через состояние: основной экран, голосовая практика, настройки.
- Все экраны построены на Jetpack Compose и Material3.

## 2. Техническое задание

### Цели
- Реализовать Android-приложение для помощи в изучении иврита.
- Добавить выбор LLM провайдера в настройках: Gemini и OpenAI.
- Позволить пользователю сохранить API ключ для выбранного провайдера.
- Хранить настройки между запусками приложения.
- Поддержать генерацию уроков по запросу, хранить прогресс локально.
- Добавить голосовую практику с распознаванием речи.

### Требования
- Платформа: Android, Kotlin, Jetpack Compose.
- Минимальный SDK: 24.
- Целевой SDK: 34.
- Хранение прогресса: Room.
- Хранение настроек: DataStore Preferences.
- HTTP-клиент: OkHttp.

### Функциональная логика
- Пользователь вводит тему урока.
- Приложение генерирует урок через LLM API выбранного провайдера.
- Запрос строится через шаблон `LlmPromptTemplates`.
- Результат сохраняется в локальную историю.
- Настройки провайдера и ключа сохраняются в DataStore.
- Если ключ не задан, используется `MockLlmService`.
- При первом запуске приложение стартует с чат-сценария знакомства и диагностики уровня.
- После диагностики приложение сохраняет профиль ученика в Room и использует его в последующих промптах.

### Принятые проектные решения
- Память об ученике хранится в Room, отдельно от истории уроков.
- На текущем этапе приложение поддерживает один локальный профиль ученика на устройстве.
- Первый запуск реализуется как первый чат-сценарий, а не как отдельная анкета.
- Базовый профиль включает имя, родной язык, интересы, область работы, цель изучения, стартовый уровень и краткое резюме диагностики.
- Любимые форматы практики и слабые места не задаются вручную на старте. Учитель выявляет их в процессе обучения и может использовать это в будущих расширениях памяти.
- LLM не сохраняет память напрямую. Она возвращает структурированный результат диагностики, а приложение извлекает его и сохраняет в Room.
- Все последующие уроки и чат-ответы строятся через промпт учителя иврита, дополненный сохраненным профилем ученика.
